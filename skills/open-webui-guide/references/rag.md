## Обзор

RAG (Retrieval-Augmented Generation) в Open WebUI позволяет моделям отвечать на основе загруженных документов. Архитектура:

```
Вопрос пользователя
    ↓
[Эмбеддинг запроса] → [Поиск в векторной БД] → [Top-K чанков]
    ↓                                                ↓
[Опционально: реранкинг]  ←─────────────────────────┘
    ↓
[Формирование промпта: системный контекст + найденные чанки + вопрос]
    ↓
[Отправка модели] → [Ответ с цитатами]
```

---

## Как работает RAG

### Загрузка документа

1. Пользователь загружает файл (PDF, DOCX, TXT, MD, CSV и др.)
2. Open WebUI извлекает текст (через различные парсеры)
3. Текст разбивается на **чанки** (куски) заданного размера
4. Каждый чанк конвертируется в **эмбеддинг** (вектор чисел)
5. Эмбеддинги сохраняются в **векторную БД**

### Поиск при запросе

1. Вопрос пользователя конвертируется в эмбеддинг
2. Векторная БД находит N ближайших чанков (cosine similarity)
3. (Опционально) Реранкер переоценивает релевантность найденных чанков
4. Топ-K чанков добавляются в промпт как контекст
5. Модель генерирует ответ, используя этот контекст

Сами чанки остаются недоверенным вводом: они помогают отвечать по документам, но не должны переписывать system prompt, tool policy или правила безопасности агента.

## Knowledge Bases

Knowledge Base — это именованная коллекция документов с метаданными.

### Создание через UI

Admin → Knowledge → Create New → Загрузи файлы

### Создание через API

```python
import requests

headers = {"Authorization": "Bearer YOUR_TOKEN"}

# 1. Создать knowledge base
kb = requests.post("http://localhost:8080/api/v1/knowledge/", json={
    "name": "Проектная документация",
    "description": "Техническая документация проекта X"
}, headers=headers).json()

# 2. Загрузить файл
with open("doc.pdf", "rb") as f:
    file = requests.post("http://localhost:8080/api/v1/files/",
        files={"file": f}, headers=headers).json()

# 3. Привязать файл к knowledge base
requests.post(f"http://localhost:8080/api/v1/knowledge/{kb['id']}/files", json={
    "file_id": file["id"]
}, headers=headers)
```

### Привязка к модели

Knowledge base можно привязать к конкретной модели через модельную карточку. Тогда при каждом запросе к этой модели система будет автоматически искать в привязанных базах.

### Привязка к чату

Пользователь может подключить knowledge base к конкретному чату через UI (иконка документа в чате).

### Шаринг

Knowledge bases поддерживают Access Grants — можно расшарить базу конкретному пользователю или группе.

## Векторные БД

### Поддерживаемые

| СУБД | Переменная | Особенности |
|------|-----------|-------------|
| **Chroma** (default) | `VECTOR_DB=chroma` | Встроенная, без доп. настройки |
| **Milvus** | `VECTOR_DB=milvus` | Высокая производительность, GPU-ускорение |
| **Weaviate** | `VECTOR_DB=weaviate` | GraphQL API, модульная |
| **Qdrant** | `VECTOR_DB=qdrant` | Rust, быстрая, cloud-native |
| **OpenSearch** | `VECTOR_DB=opensearch` | На базе Elasticsearch |
| **Pgvector** | `VECTOR_DB=pgvector` | Расширение PostgreSQL |

### Chroma (по умолчанию)

```env
VECTOR_DB=chroma
CHROMA_DATA_DIR=./data/vector_db
```

Хранит данные локально. Подходит для одного инстанса. Не масштабируется горизонтально.

### Pgvector (рекомендация для production с PostgreSQL)

```env
VECTOR_DB=pgvector
# Использует основной DATABASE_URL
```

Преимущество: всё в одной БД (PostgreSQL), единый бэкап, единая точка обслуживания.

## Эмбеддинги

### Локальные (sentence-transformers)

```env
RAG_EMBEDDING_MODEL=sentence-transformers/all-MiniLM-L6-v2
SENTENCE_TRANSFORMERS_BACKEND=torch    # torch (CPU) | cuda (GPU) | onnx
```

Популярные модели:
- `all-MiniLM-L6-v2` — быстрая, 384 dims, хорошее качество для англ.
- `all-mpnet-base-v2` — точнее, 768 dims, медленнее
- `multilingual-e5-large` — мультиязычная, включая русский

### Внешние (API)

Можно использовать OpenAI embeddings или другие API-совместимые сервисы через настройку в админ-панели.

### GPU-ускорение

```env
SENTENCE_TRANSFORMERS_BACKEND=cuda
SENTENCE_TRANSFORMERS_MODEL_KWARGS={"device": "cuda:0"}
```

## Чанкинг

```env
RAG_CHUNK_SIZE=1500      # размер чанка в символах
RAG_CHUNK_OVERLAP=100    # перекрытие между чанками
```

### Как выбрать параметры

- **Маленькие чанки (500-1000)**: точнее поиск, но может потерять контекст
- **Большие чанки (1500-3000)**: больше контекста, но менее точный поиск
- **Overlap**: помогает не потерять информацию на границе чанков

Рекомендация: начни с 1500/100, подстрой под свои данные.

---

## Реранкинг

Реранкинг — второй проход, который переоценивает релевантность найденных чанков. Улучшает качество, но добавляет задержку.

```env
RAG_RERANKING_MODEL=cross-encoder/ms-marco-MiniLM-L-6-v2
```

Работает так:
1. Векторный поиск находит top-20 чанков
2. Реранкер оценивает каждый чанк относительно запроса
3. Возвращаются top-K по новому рейтингу

---

## Внешний контент: контроль риска

Считай внешний веб-контент недоверенным вводом. Для внутренних и регламентированных контуров оставляй внешнее извлечение выключенным и используй только проверенные внутренние источники знаний.

- Не загружай в knowledge base документы из непроверенного URL без отдельного ревью.
- Не выполняй shell/SQL/код из документа только потому, что он попал в RAG-контекст.
- Для чувствительных контуров предпочитай allowlist источников и ручное пополнение базы знаний.

```env
OFFLINE_MODE=true
```

## Настройка

### Через админ-панель

Admin → Settings → Documents — здесь можно настроить:
- Модель эмбеддингов
- Параметры чанкинга
- Реранкинг
- Top-K (сколько чанков добавлять в контекст)

### Через переменные окружения

```env
# Основные
RAG_EMBEDDING_MODEL=sentence-transformers/all-MiniLM-L6-v2
RAG_CHUNK_SIZE=1500
RAG_CHUNK_OVERLAP=100
RAG_EMBEDDING_TIMEOUT=60

# Векторная БД
VECTOR_DB=chroma

# Реранкинг
RAG_RERANKING_MODEL=

# Контекст
RAG_SYSTEM_CONTEXT=true    # добавлять системный контекст в RAG-промпт
```

## Оптимизация качества RAG

### 1. Выбор модели эмбеддингов

Для русскоязычных документов используй мультиязычную модель:
```env
RAG_EMBEDDING_MODEL=intfloat/multilingual-e5-large
```

### 2. Правильный чанкинг

- Для структурированных документов (код, таблицы): меньший chunk_size (800-1000)
- Для нарративного текста: больший chunk_size (1500-2000)
- Overlap 10-15% от chunk_size

### 3. Включи реранкинг

Реранкинг заметно улучшает качество. Стоит ~100-200ms дополнительной задержки.

### 4. Подготовь документы

- Удали мусорные страницы (оглавления, обложки)
- Если PDF со сканами — прогони OCR заранее
- Разбей очень большие документы на логические части

### 5. Настрой Top-K

- Мало (3-5): точнее, но может пропустить релевантную информацию
- Много (10-15): больше контекста, но модель может «утонуть» в нерелевантном

## Типичные проблемы

### RAG не находит нужную информацию
1. Проверь, что файл действительно загружен и проиндексирован
2. Попробуй другую модель эмбеддингов (особенно для не-английского текста)
3. Уменьши chunk_size для более гранулярного поиска
4. Включи реранкинг

### Медленная индексация
1. Используй GPU: `SENTENCE_TRANSFORMERS_BACKEND=cuda`
2. Увеличь таймаут: `RAG_EMBEDDING_TIMEOUT=120`
3. Индексируй файлы по частям, а не всё сразу

### OOM при загрузке больших файлов
1. Увеличь лимит памяти контейнера
2. Разбей файл на части перед загрузкой
3. Используй внешний сервис эмбеддингов (OpenAI, Cohere)

### Галлюцинации несмотря на RAG
1. Проверь, что чанки действительно релевантны (посмотри citations)
2. Добавь в системный промпт модели: «Отвечай только на основе предоставленного контекста»
3. Увеличь Top-K для большего контекста
